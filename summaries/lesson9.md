This is the first lesson of Part 2 of "Practical Deep Learning for Coders" — numbered Lesson 9 because Part 1 had eight lessons. Jeremy half-jokingly calls it "Impractical Deep Learning for Coders," because while Part 1 focused on practical applications, Part 2 dives into foundational details and generative model fun. The goal is not just to use deep learning but to understand it deeply enough to do research, customize production systems, and keep up as the field evolves at breakneck speed.

The lesson splits into two halves. The first half is a hands-on tour of Stable Diffusion using Hugging Face's Diffusers library. Jeremy walks through the `stable_diffusion.ipynb` notebook from the `diffusion-nbs` repo, showing how to load a pretrained Stable Diffusion pipeline with `from_pretrained()`, pass in a text prompt like "a photograph of an astronaut riding a horse," and get back a generated image in about 30 seconds and 51 inference steps. He explains that setting a manual random seed makes outputs reproducible and shareable.

From there, the lesson explores several powerful tweaks available through Diffusers. **Guidance scale** controls how strongly the model attends to the prompt versus just generating a generic image — low values (~1) produce incoherent results, the default of 7.5 produces good results, and very high values (~14) can become overly abstract. **Negative prompts** let you subtract concepts: generating "a Labrador in the style of Vermeer" with a negative prompt of "blue" removes blue tones from the output. The **image-to-image pipeline** lets you start the diffusion process from a noisy version of an existing sketch or image rather than pure noise, preserving the composition while refining quality — the **strength** parameter controls how much freedom the model has to deviate from the original. Jeremy shows how Pedro Cuenca chained outputs through a second pass to produce Van Gogh–style wolf paintings from rough compositions.

The lesson then covers fine-tuning approaches. Justin at Lambda Labs fine-tuned Stable Diffusion on a Pokémon dataset (with captions auto-generated by an image captioning model), producing a model that generates novel Pokémon from arbitrary prompts. **Textual inversion** is a lighter approach: you train a single new token embedding (e.g., "watercolor-portrait") on just a few example images, then use it in prompts like "woman reading in the style of \<watercolor-portrait\>." **Dreambooth** is similar but fine-tunes the whole model to associate a rarely-used token (like "sks") with a specific subject — Pedro used it to generate paintings of Jeremy Howard in various styles. Jeremy shares his own attempt at textual inversion with his daughter's oversized teddy bear "Tiny," noting the imperfect but instructive results.

The second half of the lesson is a from-scratch conceptual explanation of how Stable Diffusion works, presented in a novel way that Jeremy emphasizes is mathematically equivalent to the standard derivation but conceptually much simpler. He begins with a thought experiment: imagine a magic API that takes any image and returns the probability that it's a handwritten digit. If such a function existed, you could use it to *generate* digits — take a noisy image, compute how changing each pixel affects the probability (the gradient), then nudge the pixels in the direction that increases the probability. Repeat this process, and noise gradually transforms into a recognizable digit.

Jeremy explains that computing this gradient pixel-by-pixel (finite differencing) is slow — 784 function calls for a 28×28 image — but PyTorch's `.backward()` gives us analytic derivatives in one pass. Moreover, we don't actually need the probability function itself; we only need the gradient — the information about which pixels to change. So rather than training a model to output probabilities, we can train a neural network to directly predict the *noise* that was added to a clean image. The training data is trivially easy to generate: take real digits, add varying amounts of random Gaussian noise, and train the network (with MSE loss) to predict the noise. At inference time, we feed pure noise into the trained model, it predicts what portion is noise, we subtract a fraction of that prediction, and repeat — each step producing something slightly more digit-like.

The specific architecture used for this noise-prediction network is the **U-Net**, originally developed for medical imaging. Its input is a somewhat noisy image (or pure noise); its output is the predicted noise. Subtracting the output from the input yields a cleaner image.

Jeremy then addresses the computational challenge: real images are 512×512×3 = 786,432 pixels, which is expensive to work with directly. The solution is an **autoencoder** — a neural network that compresses images through successive stride-2 convolutions (the **encoder**) down to a compact 64×64×4 representation (16,384 values — a 48× reduction), then reconstructs them through inverse convolutions (the **decoder**). Trained with MSE loss between input and output, the autoencoder learns to preserve all essential information in the compressed form. This compressed representation is called the **latents**, and it's what the U-Net actually operates on — making training and inference dramatically faster. The autoencoder is called the **VAE** (Variational Autoencoder); only the decoder is needed at inference time.

To make the model generate *specific* images from text prompts, Jeremy explains **guidance**. For MNIST, you could one-hot encode the digit label and pass it alongside the noisy input — the model learns to predict noise conditioned on knowing what digit it's looking at, so at inference time, passing "3" plus noise produces a three. For natural language, one-hot encoding every possible sentence is impossible, so instead we use **CLIP** (Contrastive Language–Image Pre-training). CLIP trains two models simultaneously — a text encoder and an image encoder — using a contrastive loss: matching image-text pairs should produce similar embeddings (high dot product), while non-matching pairs should produce dissimilar ones. The result is a text encoder that maps semantically similar sentences to nearby points in embedding space, giving us the rich guidance signal needed for text-to-image generation.

The lesson concludes with a discussion of **samplers** and an intriguing analogy to deep learning optimizers. The inference process — repeatedly predicting and subtracting noise — mirrors SGD: the constant multiplied by the predicted noise is analogous to a learning rate, and concepts like momentum and Adam have direct parallels in differential equation solvers. Jeremy and Johno's early research suggests that reframing diffusion as an optimization problem (rather than a differential equation) opens up promising new directions, including using perceptual loss instead of MSE, and potentially eliminating the need to pass the "time step" *t* into the model at all. The term "time steps" itself is a confusing historical artifact — it refers to a noise schedule index, not actual time — and Jeremy suggests it may become unnecessary as the field evolves.

Jeremy closes by previewing what's next: finishing the pipeline code walkthrough, then rewinding all the way to pure Python to rebuild everything from scratch — convolutions, autoencoders, U-Nets, CLIP, and all — building up the foundational knowledge that makes all these rapidly changing details comprehensible.

- **Potential Research Directions**
  - Treating diffusion inference as an optimization problem rather than a differential equation solving problem — applying optimizer concepts (learning rate, momentum, Adam) to the sampling process
  - Removing the time step *t* as an input to the U-Net, since the model should be able to infer noise level from the input itself
  - Using perceptual loss instead of MSE for training the noise prediction model
  - Using the denoised output directly rather than adding noise back at each step
  - Exploring whether sophisticated loss functions improve generation quality
  - Exploring connections between diffusion samplers and deep learning optimizers (early results from Johno are "extremely positive")
